# Spring Cloud / Kubernetes 服务发布问题 After

这次线上问题出在滚动发布。服务是 Spring Cloud，跑在 Kubernetes 上，注册中心用 Nacos。发布新版本时，Pod 已经显示 Ready，但网关还有一小段时间会把请求打到没有完成注册同步的实例上，结果出现了几次 502。

排查顺序很直接。先看应用日志，新 Pod 启动后还在加载配置和初始化连接池；再看 Kubernetes 事件，readinessProbe 通过得太早；最后看网关日志，502 集中在新 Pod Ready 后的 20 秒内。也就是说，问题不是代码接口变慢，而是“容器 Ready”和“服务真的能接流量”之间有时间差。

我们先改了 readinessProbe，把检查条件从端口存活改成应用健康接口，并在健康接口里确认 Nacos 注册状态。这个方案能挡住大部分提前接流量的问题，但还有代价：启动时间会增加十几秒，发布总耗时变长。下一步需要补一条发布监控，专门看新旧 Pod 切换期间的 5xx，而不是只看发布是否成功。
