# 技术博客专项规则

技术博客去 AI 味的核心不是变口语，而是把“正确但空”的说明，改成“可复现、有边界、有取舍”的经验。

技术文章可以正式，可以有术语，可以有结构。问题通常出在：先讲宏大背景、再夸工具强大、再列核心特性、最后总结潜力，但读者看不到环境、错误、代码、代价和失败路径。

## 保留什么

- 准确技术名词
- 版本号、运行环境、依赖条件
- 错误日志、异常现象、复现步骤
- 代码片段和配置差异
- 排查顺序
- 方案取舍
- 不适用场景
- 仍未解决的问题

## 删除什么

- 在现代软件开发中
- X 是一个强大的工具
- 极大提升开发效率
- 提供了丰富能力
- 被广泛应用于各种场景
- 本文将介绍核心特性与最佳实践
- 让我们一起深入了解
- 通过本文你将掌握
- 总的来说，X 具有巨大潜力

## 技术博客的好开头

优先从这几类入口开始：

- 一个明确问题：`滚动发布时，新 Pod Ready 后仍然出现 502。`
- 一个具体环境：`Spring Boot 3.2 + Kubernetes 1.29 + Nacos 2.3。`
- 一个错误日志：`CrashLoopBackOff 不是内存不够，而是启动脚本找不到配置文件。`
- 一个取舍：`最后我没有用 operator，因为我们只需要每天同步一次配置。`
- 一个限制：`这个方案只适合单集群，多集群会遇到权限和命名冲突。`

不要从“时代背景”开始。读者打开技术文章，不需要先被提醒软件开发很重要。

## 常见改法

### 1. 把工具介绍改成使用边界

差：

Redis 是一个高性能的内存数据库，在现代互联网系统中被广泛使用。

好：

这次我只用 Redis 做一件事：给短信验证码加 5 分钟过期时间。它适合这个场景，因为数据短、读写频繁、丢了也能重发。

### 2. 把最佳实践改成踩坑路径

差：

下面介绍 Kubernetes 服务发布的最佳实践。

好：

这次发布失败不是镜像问题，而是 readinessProbe 放得太宽。Pod 显示 Ready 时，应用还没完成注册中心同步。

### 3. 把“效率提升”改成具体节省

差：

该脚本显著提升了部署效率。

好：

原来每次发布要手动改 4 个配置文件，现在脚本从 `.env` 生成配置，少了 3 次复制粘贴。

### 4. 把“系统稳定性”改成失败条件

差：

该方案可以提升系统稳定性。

好：

这个方案能挡住接口超时后的重复请求，但挡不住数据库已经写入、响应丢失的情况。

### 5. 把“深入理解”改成读者会得到什么

差：

本文将带你深入理解 Spring Cloud Gateway 的核心原理。

好：

这篇文章只讲一个点：Gateway 路由刷新为什么会比服务注册慢半拍，以及怎么确认慢在哪里。

## 技术文章检查清单

发布前问自己：

- 开头前三句是否出现真实问题？
- 环境和版本是否足够复现？
- 有没有错误日志或异常现象？
- 有没有写排查顺序，而不是只给最终答案？
- 有没有说明方案为什么有效？
- 有没有说明代价和不适用场景？
- 结尾是否停在具体结论，而不是“未来值得持续探索”？

## 不要误伤

这些情况可以保留正式表达：

- 官方文档式教程需要概念定义。
- 面向新人时可以先解释术语。
- 架构设计文档可以用较稳定的结构。
- 性能测试报告可以使用固定指标。

但即使是正式技术文本，也要尽量写清楚：对象、条件、结果、限制。
