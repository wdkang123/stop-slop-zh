# 技术博客专项 before/after

这组示例用于扩充 v0.2.0 的技术博客专项。重点不是把技术文章写得更“轻松”，而是去掉空泛介绍，补回问题、环境、排查、取舍和边界。

## Example 1: Kubernetes 滚动发布

Before:

在现代云原生架构中，Kubernetes 已经成为服务部署和管理的重要平台。滚动发布作为一种常见的发布方式，可以有效保障服务升级过程中的稳定性和连续性。本文将从滚动发布的核心机制出发，分析其在实际业务中的应用价值和最佳实践。

Problem:

- 宏大背景开头
- “应用价值和最佳实践”空泛
- 没有具体故障

After:

这次发布的问题出在滚动发布的前 30 秒。新 Pod 已经 Ready，网关开始转发请求，但应用还没完成注册中心同步，结果出现了几次 502。下面只记录排查过程：怎么确认不是镜像问题，怎么定位到 readinessProbe，以及最后为什么把健康检查改到应用层。

Why better:

- 从故障进入
- 保留 Kubernetes 术语
- 明确文章范围

## Example 2: Redis 缓存穿透

Before:

Redis 是一个高性能、灵活、强大的缓存工具，在互联网系统中有着非常广泛的应用。合理使用 Redis 可以显著提升系统性能，降低数据库压力。本文将介绍 Redis 缓存穿透问题及其解决方案。

Problem:

- 工具介绍模板
- 空泛形容词
- 没有业务场景

After:

我们遇到的缓存穿透很具体：商品详情页被刷不存在的 `skuId`，Redis 查不到，MySQL 也查不到，但请求会一直打到数据库。最后用了两层处理：不存在的 `skuId` 缓存空值 60 秒，明显非法的参数先过布隆过滤器。空值缓存简单，但会占一点内存；布隆过滤器省数据库，但要接受少量误判。

Why better:

- 写清请求对象
- 说明方案和代价
- 没有泛泛介绍 Redis

## Example 3: Spring Boot 启动慢

Before:

随着业务复杂度的不断提升，Spring Boot 应用的启动性能逐渐成为开发者关注的重要问题。如何优化启动速度、提升开发效率，是每个团队都需要持续探索的方向。

Problem:

- “业务复杂度提升”铺垫
- 没有版本、环境、耗时
- “持续探索”无结论

After:

这个 Spring Boot 服务启动从 18 秒涨到 52 秒，版本是 3.2.4，问题出现在本地开发环境。用 `ApplicationStartup` 看了一遍，最慢的不是 Bean 初始化，而是启动时连了 6 个外部服务。我们最后把 4 个非必要连接改成懒加载，本地启动回到 24 秒。线上没这么改，因为线上需要启动时提前失败。

Why better:

- 有数字和版本
- 区分本地与线上
- 写出取舍

## Example 4: Docker 镜像过大

Before:

Docker 镜像体积优化是容器化实践中的重要环节。一个高质量的镜像不仅可以提升部署效率，也可以降低资源消耗。本文将从多个维度分析镜像优化的方法。

Problem:

- “重要环节”空话
- 没有当前体积和目标
- “多个维度”模板化

After:

我们的后端镜像从 1.2GB 降到 310MB，主要靠两步：把构建阶段和运行阶段拆开，只把 jar 包和运行时依赖放进最终镜像；再把基础镜像从完整 JDK 换成 JRE。代价是排查线上问题时少了一些调试工具，所以保留了一个 debug tag，只在临时排障时用。

Why better:

- 有前后对比
- 有具体做法
- 有代价

## Example 5: 前端构建缓存

Before:

在前端工程化不断发展的背景下，构建速度已经成为影响团队研发效率的重要因素。通过合理利用缓存能力，可以有效提升构建效率，优化开发体验。

Problem:

- 背景铺垫
- “有效提升”没有数据
- 没有工具和场景

After:

CI 上的前端构建原来要 11 分钟，主要慢在 `pnpm install` 和 Vite build。我们把 pnpm store 缓存在 GitHub Actions 里，又把 `node_modules/.vite` 从缓存里拿掉，因为它经常因为依赖变动失效。现在稳定在 6 分钟左右。这个方案只适合依赖变化不频繁的仓库。

Why better:

- 有工具和数字
- 写出错误尝试
- 有适用条件

## Example 6: 数据库索引

Before:

索引是数据库性能优化中非常重要的手段。合理设计索引可以显著提升查询效率，降低数据库压力。本文将介绍索引设计的基本原则和常见优化方法。

Problem:

- 教材式开头
- 没有具体查询
- 结论太泛

After:

这次慢查询不是缺索引，而是索引顺序错了。查询条件是 `tenant_id + status + created_at`，原索引把 `created_at` 放在最前面，导致同一个租户下的状态筛选没用上。调整成 `(tenant_id, status, created_at)` 后，查询从 1.8 秒降到 90ms。这个结论只适用于当前筛选条件，不是所有时间范围查询都该这么建。

Why better:

- 有字段和索引顺序
- 有性能对比
- 不扩大结论

## Example 7: API 超时

Before:

在微服务系统中，接口超时是一个常见且复杂的问题。它不仅会影响用户体验，也会对系统稳定性造成一定挑战。因此，我们需要从多个层面进行分析和优化。

Problem:

- “常见且复杂”空泛
- 没有调用链
- “多个层面”模板化

After:

这次超时发生在订单服务调库存服务。订单接口设置 2 秒超时，但库存服务最慢会查 3 个仓库，峰值时要 2.6 秒。我们先把库存查询改成并发，再把订单侧超时调到 3 秒。没有继续调大，因为支付页超过 3 秒用户就会重复点击。

Why better:

- 有调用关系
- 有时间数据
- 有产品层面的边界

## Example 8: 日志字段混乱

Before:

日志系统是保障系统可观测性的重要基础。通过统一日志规范，可以帮助团队更加高效地定位问题，提升系统运维能力。

Problem:

- 运维套话
- 没有说明哪里混乱
- 没有执行方式

After:

我们的问题不是没有日志，而是字段名不一致。同一个请求 ID，有的服务叫 `traceId`，有的叫 `request_id`，还有一个老服务写在 message 里。排查跨服务问题时，Kibana 里要查三次。先不改所有日志格式，第一步只做一件事：网关统一写 `traceId`，下游服务沿用这个字段。

Why better:

- 写出具体混乱点
- 有渐进方案
- 不假装一次解决全部问题

## Example 9: 消息队列重试

Before:

消息队列在分布式系统中承担着削峰填谷和异步解耦的重要作用。为了保障消息消费的可靠性，我们需要建立完善的重试机制和异常处理能力。

Problem:

- 概念定义过多
- “完善机制”没有边界
- 没有失败后果

After:

订单创建后会发一条消息给积分服务。问题是积分服务偶尔超时，消息被重试 5 次后进死信队列，但没人看死信队列，用户第二天才发现积分没到账。我们改了两处：消费失败超过 3 次就发告警；补一个每天跑一次的死信补偿任务。它不能保证实时到账，但至少不会静默丢掉。

Why better:

- 有业务后果
- 有具体机制
- 写清仍然存在的限制

## Example 10: TypeScript 类型收窄

Before:

TypeScript 提供了强大的类型系统，可以帮助开发者在编译阶段发现潜在问题。掌握类型收窄技巧，有助于提升代码质量和开发效率。

Problem:

- 工具能力空泛
- 没有代码场景
- “提升质量”没有说明

After:

这次 bug 是一个 `string | null` 被当成 `string` 传进了 URL 拼接。我们没有用 `as string` 压过去，而是在取值后先判断空值，并把后续逻辑放进同一个分支。TypeScript 的类型收窄在这里有用，因为它逼着你处理“没有值”的路径，而不是让错误到运行时才暴露。

Why better:

- 有真实 bug 类型
- 说明为什么不用断言
- 保留 TypeScript 价值但不空泛
